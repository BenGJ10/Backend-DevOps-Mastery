# Docker Images and Containers

## Overview

In Docker, **images** and **containers** are closely related but serve different purposes.  

An image defines **what should run**, while a container represents **a running instance of that definition**.

---

## Docker Images

### What Is a Docker Image?

A **Docker image** is an **immutable, read-only template** that contains everything required to run an application, including:
- application code
- runtime
- libraries and dependencies
- environment configuration
- startup instructions

Images do not execute by themselves; they are used to create containers.

---

### Key Properties of Docker Images

- Immutable once built
- Layered architecture
- Versioned using tags
- Reusable across environments

Because images are immutable, changes require building a **new image version**.

---

### Image Layers

Docker images are built as a stack of layers.

Each instruction in a Dockerfile creates a new layer, which allows:
- caching
- faster builds
- efficient storage

Unchanged layers are reused across builds.

---

## Docker Containers

### What Is a Docker Container?

A **Docker container** is a **running instance of a Docker image**.  
It represents an isolated execution environment where the application actually runs.

Multiple containers can be created from the same image.

---

### Key Properties of Containers

- Ephemeral by design
- Isolated from other containers
- Have their own filesystem, network, and process space
- Can be started, stopped, and destroyed

Containers should be replaceable, not manually modified.

---

## Image vs Container

| Aspect | Docker Image | Docker Container |
|------|--------------|------------------|
| Nature | Blueprint | Running instance |
| Mutability | Immutable | Mutable at runtime |
| Execution | Cannot run | Actively running |
| Lifecycle | Built once | Created, started, stopped |
| Usage | Packaging | Execution |

---

## Typical Workflow

1. Write application code

2. Create Dockerfile

3. Build image

4. Run container from image

5. Stop and remove container when done

This workflow enables repeatable and predictable deployments.

---

## Backend Engineering Perspective

In backend systems:

- images define service versions
- containers run services in environments
- CI pipelines build images
- deployment systems run containers
- scaling is done by running more containers

This separation enables clean, controlled deployments.

---

## Interview Questions

### 1. What is a Docker image?
**Answer:**  
A Docker image is an immutable template that contains everything needed to run an application.

---

### 2. What is a Docker container?
**Answer:**  
A Docker container is a running instance of a Docker image.

---

### 3. Can multiple containers be created from the same image?
**Answer:**  
Yes, multiple independent containers can be started from a single image.

---

### 4. Why are Docker images immutable?
**Answer:**  
Immutability ensures consistency, reproducibility, and reliable deployments.

---

### 5. What happens to a container when it stops?
**Answer:**  
The container stops executing but its filesystem state remains until it is removed.

---

## Summary

- Docker images define how applications should run

- Docker containers are runtime instances of images

- Images are immutable and versioned

- Containers are ephemeral and replaceable

- This separation enables scalable backend deployments
