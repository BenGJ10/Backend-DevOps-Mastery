# Pushing Code to GitHub

## Overview

**Pushing** in Git is the operation of **sending local commits to a remote repository** such as GitHub.  
It publishes your work so that it becomes visible to teammates, CI/CD pipelines, deployment systems, and backup storage.

Until you push, your commits exist **only on your local machine**.  
Pushing makes your branch available remotely and synchronizes local and remote histories.

---

## Prerequisites Before Pushing

Before pushing, you should already have:

- a **local Git repository**
- at least one **commit**
- a **remote repository URL** (e.g., GitHub repo)
- configured branch name (`main`, `develop`, or feature branch)

---

## Step 1 — Add Remote Repositorys

Connect your local repo to a GitHub repo:

```bash
git remote add origin <remote-repository-url>
```

Example:

```bash
git remote add origin https://github.com/username/project.git
```

Verify remotes:

```bash
git remote -v
```

---

## Step 2 — Push for the First Time

Push current branch to remote and set upstream:

```bash
git push -u origin main
```

What `-u` does:

* links local branch with remote branch
* allows using simple `git push` later

---

## Step 3 — Regular Push After Changes

Typical workflow:

```bash
git add .
git commit -m "Implement user login API"
git push
```

Because upstream is set, no need to specify branch each time.

---

## Pushing a New Branch

If the branch does not exist on GitHub yet:

```bash
git push -u origin feature-auth
```

This:

* creates remote branch
* tracks it with your local branch

---

## Viewing Remote Branches

```bash
git branch -a
```

Shows local and remote branches.

Remote branches appear as:

```text
remotes/origin/main
remotes/origin/feature-auth
```

---

## Handling Push Rejections (Common Case)

If someone pushed new commits before you, you may see:

```text
! [rejected] main -> main (non-fast-forward)
error: failed to push some refs
```

This happens due to **out-of-date branch history**.

### Fix: Pull before pushing

```bash
git pull --rebase origin main
git push
```

Rebase keeps history linear.

> Avoid `--force` unless you fully understand consequences.

---

## Force Push (Use with Extreme Caution)

Force push rewrites remote history:

```bash
git push --force
```

or safer variant (keeps others’ work):

```bash
git push --force-with-lease
```

Use only when:

* rewriting your own branch history
* correcting mistakenly pushed commits
* in non-shared feature branches

Never use on protected branches like `main`.

---

## Push Tags to GitHub

### Push a specific tag

```bash
git push origin v1.0.0
```

### Push all tags

```bash
git push --tags
```

Tags are commonly used for **releases**.

---

## Practical Backend Engineering Use Cases

Pushing to GitHub is required when:

* CI/CD pipelines should run
* deployments are triggered from GitHub
* teammates need latest code
* creating pull requests
* mirroring repository for backup

In many organizations, **nothing deploys unless code is pushed**.

## Interview Questions

### 1. What does `git push` do?

**Answer:**
`git push` sends local commits to a remote repository such as GitHub, updating the remote branch with local changes.

---

### 2. What is the purpose of `-u` in `git push -u origin main`?

**Answer:**
It sets the **upstream branch**, linking your local branch with the remote branch so future pushes and pulls can omit branch names.

---

### 3. Why might a push be rejected?

**Answer:**
Push is rejected when the remote branch contains commits not present locally (non-fast-forward). You need to `git pull` first to synchronize histories.

---

### 4. What is the difference between `git push` and `git pull`?

**Answer:**

* `git push` uploads local commits to remote
* `git pull` downloads commits from remote and integrates them into the local branch

---

### 5. When is `git push --force` dangerous?

**Answer:**
It rewrites remote history and can delete others’ work. It should not be used on shared branches like `main` and only in controlled scenarios.

---

## Summary

* Pushing publishes local commits to GitHub

* Remote must be added using `git remote add origin`

* `-u` sets branch tracking for future pushes

* Push rejections occur due to out-of-date branches

* Force push rewrites history and must be used cautiously

* Pushing enables collaboration, CI/CD, and deployment pipelines
