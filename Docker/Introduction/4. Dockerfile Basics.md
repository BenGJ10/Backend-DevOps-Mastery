# Dockerfile Basics

A **Dockerfile** is a text document that contains a set of instructions used to **build a Docker image**.  

It defines the application environment, dependencies, configuration, and startup behavior in a reproducible way.

Dockerfiles enable teams to **standardize builds**, eliminate manual setup, and ensure that applications run consistently across all environments.

---

## Why Dockerfiles Matter

Without Dockerfiles:

- environments must be configured manually  
- builds become inconsistent  
- onboarding slows down  
- deployments become risky  

Dockerfiles solve this by making infrastructure **declarative and version-controlled**.

They act as the **blueprint for your container**.

---

## Basic Dockerfile Structure

**java** Example:

```dockerfile
FROM openjdk:17
WORKDIR /app
COPY target/app.jar app.jar
EXPOSE 8080
CMD ["java", "-jar", "app.jar"]
```

**Python** Example:

```dockerfile
FROM python:3.9
WORKDIR /app
COPY . .
RUN pip install -r requirements.txt
EXPOSE 5000
CMD ["python", "app.py"]
```
---

## Core Dockerfile Instructions

### `FROM`

Specifies the base image.

```dockerfile
FROM node:18
```

Every Dockerfile must start with a base image unless building from scratch.

**Best Practice:** Use minimal and trusted images.

---

### `WORKDIR`

Sets the working directory inside the container.

```dockerfile
WORKDIR /app
```

All subsequent commands execute relative to this path.

Avoid using repeated `cd` commands.

---

### `COPY`

Copies files from the host machine into the container.

```dockerfile
COPY . .
```

Commonly used to copy:

* source code
* compiled binaries
* configuration files

Prefer copying only necessary files to keep images small.

---

### `RUN`

Executes commands during image build.

```dockerfile
RUN apt-get update && apt-get install -y curl
```

Typical uses:

* installing dependencies
* downloading packages
* setting up runtime tools

Each `RUN` creates a new layer.

---

### `EXPOSE`

Documents the port the container listens on.

```dockerfile
EXPOSE 8080
```

It does not publish the port — it serves as metadata.

This is optional as we can publish ports at runtime with `docker run -p`.

---

### `CMD`

Defines the default command executed when the container starts.

```dockerfile
CMD ["node", "server.js"]
```

Only one `CMD` should exist; the last one overrides previous entries.

---

## Build an Image from a Dockerfile

Run:

```bash
docker build -t backend-app:1.0 .
```

Explanation:

* `-t` assigns a tag
* `.` points to the build context

Docker processes instructions from top to bottom.

---

## Layer Caching Behavior

Docker caches each instruction.

If nothing changes in a layer:

✅ it is reused

❌ it is not rebuilt

Place stable steps early and frequently changing ones later.

---

## Example Optimized Dockerfile

```dockerfile
FROM node:18

WORKDIR /app

COPY package.json .
RUN npm install

COPY . .

CMD ["npm", "start"]
```

Dependency installation is cached unless `package.json` changes.

---

## `.dockerignore` (Highly Recommended)

Prevents unnecessary files from entering the image.

Example:

```
node_modules
.git
logs
.env
```

Benefits:

* smaller images
* faster builds
* improved security

---

## Common Mistakes

### Using Large Base Images

Leads to slow pulls and higher storage usage.

---

### Copying Everything

Increases attack surface and image size.

---

### Too Many Layers

Impacts performance and caching.

---

### Hardcoding Secrets

Never include credentials in Dockerfiles.

Use environment variables or secret managers.

## Interview Questions

### 1. What is a Dockerfile?

**Answer:**
A Dockerfile is a script containing instructions to build a Docker image.

---

### 2. Which instruction defines the base image?

**Answer:**
`FROM`.

---

### 3. What is the difference between `RUN` and `CMD`?

**Answer:**
`RUN` executes during build, while `CMD` runs when the container starts.

---

### 4. Why is `.dockerignore` important?

**Answer:**
It prevents unnecessary files from being included in the image.

---

### 5. How do you build an image from a Dockerfile?

**Answer:**
Using `docker build -t <image-name> .`.

---

## Summary

* Dockerfiles define how images are built

* They provide reproducible and automated environments

* Instructions are executed in layers

* Proper structure improves build speed and security

* Dockerfiles are foundational to modern backend deployment workflows
