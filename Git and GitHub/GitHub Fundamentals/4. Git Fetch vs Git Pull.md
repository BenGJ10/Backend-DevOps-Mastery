# Git Fetching and Pulling

## Overview

In collaborative development, your local repository must regularly synchronize with the **remote repository** (e.g., GitHub).  
Git provides two primary commands for this:

- `git fetch` — **downloads changes without merging**
- `git pull` — **downloads and automatically integrates changes**

Understanding the difference between these two is critical to avoid accidental overwrites, merge conflicts, or unexpected code changes in active backend projects.

---

## `git fetch` — Download Without Integration

### Definition

`git fetch` retrieves the latest commits, branches, and tags from the remote repository **but does not modify your working directory or current branch**.

Your local files remain unchanged.

---

### Basic Usage

```bash
git fetch
```

Fetches updates for the default remote (`origin`).

Fetch a specific branch:

```bash
git fetch origin main
```

---

### What Happens Internally

* remote commits are downloaded
* remote tracking branches (e.g., `origin/main`) are updated
* **your local branch is unchanged**

This allows **safe inspection** before merging.

---

### When to Use `git fetch`

Use fetch when you want to:

* preview changes before merging
* see what teammates pushed
* avoid disrupting your working directory
* update references for comparisons/diffs

Example inspection:

```bash
git diff main origin/main
```

---

## `git pull` — Download and Integrate

### Definition

`git pull` retrieves remote changes **and integrates them into your current branch**.

It is equivalent to:

```bash
git fetch
git merge
```

(or `rebase`, depending on config).

---

### Basic Usage

```bash
git pull
```

Pull for a specific branch:

```bash
git pull origin main
```

Your working directory is updated with merged changes.

---

## Merge vs Rebase During Pull

`git pull` default behavior = fetch + merge.

Example:

```bash
git pull --rebase
```

Replays your commits on top of remote history, resulting in:

* cleaner linear history
* same final content
* different commit graph

Teams decide policy based on workflow.

---

## Practical Example Workflow

You and teammate work on `main`.

* teammate pushes commits
* your branch is now behind

You run:

```bash
git pull
```

Now your local branch:

* contains your commits
* includes teammate’s commits
* may require conflict resolution

---

## Viewing Remote Tracking Branches

```bash
git branch -r
```

Shows branches like:

```text
origin/main
origin/dev
```

These are **read-only pointers** updated during fetch/pull.

---

## Handling Conflicts During Pull

If both you and remote modified same lines:

* Git stops merge/rebase
* conflicting files are marked
* you must resolve conflicts manually
* then continue:

```bash
git add .
git merge --continue
```

or for rebase:

```bash
git rebase --continue
```

---

## Key Differences Between Fetch and Pull

| Feature                              | `git fetch`                       | `git pull`                |
| ------------------------------------ | --------------------------------- | ------------------------- |
| Downloads remote updates             | Yes                               | Yes                       |
| Automatically updates current branch | No                                | Yes                       |
| Safe, non-destructive                | Yes                               | Can cause conflicts       |
| Typical use case                     | Review first                      | Auto-sync                 |
| Internals                            | Only updates remote-tracking refs | Fetch + merge (or rebase) |

---

## Backend Engineering Perspective

Fetch vs pull matters when:

* multiple developers commit to the same service repo
* production hotfixes are merged while you are working
* CI pipelines rely on up-to-date branches
* avoiding merge conflicts is important
* investigating incoming changes before updating working tree

Teams often recommend:

* frequent **fetch**
* careful **pull with rebase**

---

## Interview Questions

### 1. What is the difference between `git fetch` and `git pull`?

**Answer:**

* `git fetch` downloads remote changes without modifying local branches
* `git pull` downloads changes and immediately integrates them into the current branch (fetch + merge or rebase)

---

### 2. When would you prefer `git fetch` over `git pull`?

**Answer:**
Prefer `fetch` when you want to **review changes first** or avoid modifying your working directory immediately.

---

### 3. What does `git pull --rebase` do?

**Answer:**
It fetches remote changes and **replays your commits on top of them**, resulting in a linear history instead of a merge commit.

---

### 4. Where are remote changes stored after `git fetch`?

**Answer:**
They are stored in **remote-tracking branches** such as `origin/main`, which can be compared or merged manually.

---

### 5. Can `git pull` cause merge conflicts? Why?

**Answer:**
Yes. If your local commits conflict with remote commits on the same lines or files, integrating them using pull may cause conflicts requiring manual resolution.

---

## Summary

* `git fetch` downloads updates without modifying your branch

* `git pull` downloads and integrates updates automatically

* Pull internally performs fetch + merge (or rebase)

* Fetch is safer for inspection, pull is faster for synchronization

* Understanding both is necessary for collaborative backend development
